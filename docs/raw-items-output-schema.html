Octopus Crawler Contract

八爪鱼最终输出表：raw_items

这张表只表达“抓到了什么内容”。运行过程、请求日志、失败重试、入口 URL、调试 payload 都不进入最终输出表，避免把爬虫内部复杂度转嫁给下游消费方。

设计结论

保留 17 个字段。content 保存主内容，context_content 保存主内容之外的结构化上下文；来源差异统一落到 metrics 与 extra。

最小输出

下游只需要知道 item 身份、来源、时间、内容和少量扩展，不需要知道 crawler 怎么抓。

内容入表

content 是最终输出的一部分。评论、引用、thread、README 等上下文进入 context_content。

JSON 收口

不同来源的指标和补充字段变化很快，用 metrics / extra 承接，稳定字段保持少而清楚。

字段取舍规则

  • 保留下游几乎每次消费都需要，或者决定去重、排序、展示、追溯。
  • 合并语义接近但不稳定的字段进入 metrics 或 extra，例如 stars、likes、feed_url、repo topics。
  • 删除只服务抓取过程的字段不进最终表，例如 fetch_status、fetch_error、raw_payload、canonical_url、run_id。

建议 MySQL DDL

CREATE TABLE raw_items (
  id CHAR(32) NOT NULL COMMENT '全局唯一 ID',
  url TEXT NOT NULL COMMENT '内容本身 URL',

  source_type VARCHAR(64) NOT NULL COMMENT '来源大类',
  sub_source_type VARCHAR(128) NOT NULL COMMENT '具体来源或采集通道',
  item_type VARCHAR(64) NOT NULL COMMENT '内容对象类型',
  author_id VARCHAR(255) NULL COMMENT '来源侧作者稳定 ID',
  author_url TEXT NULL COMMENT '作者主页 URL',

  created_date DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3) COMMENT '首次写入时间 UTC',
  updated_date DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3)
    ON UPDATE CURRENT_TIMESTAMP(3) COMMENT '最后更新时间 UTC',
  source_published_date DATETIME(3) NULL COMMENT '来源发布时间 UTC',
  snapshot_date DATE NOT NULL COMMENT '业务归属日期',

  title TEXT NOT NULL COMMENT '标题',
  metrics JSON NULL COMMENT '来源原生指标',
  content MEDIUMTEXT NULL COMMENT '主内容，下游默认消费文本',
  context_content JSON NULL COMMENT '主内容之外的上下文内容',
  extra JSON NULL COMMENT '来源特定扩展字段',
  scrape_config_snapshot JSON NULL COMMENT '抓取配置快照',

  PRIMARY KEY (id),
  KEY idx_snapshot_date (snapshot_date),
  KEY idx_source_snapshot (source_type, sub_source_type, snapshot_date),
  KEY idx_item_type_snapshot (item_type, snapshot_date),
  KEY idx_source_published_date (source_published_date)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

现有来源套用

GitHub Trending / Search

source_type=code_host, sub_source_type=github_trending/github_search, item_type=repo。content 放 README 主文本，context_content 放 description、topics、图片等上下文。

HackerNews

source_type=community, sub_source_type=hackernews, item_type=article。外链正文放 content，高赞评论和讨论页放 context_content。

X / Twitter

source_type=social, sub_source_type=x_timeline/x_search, item_type=tweet。推文文本放 content，引用推文和 thread 放 context_content。

RSS / 官方博客

source_type=website, sub_source_type=rss/html_blog, item_type=article。正文放 content，RSS 摘要、feed URL、原始 guid 放 context_content 或 extra。

Product Hunt

source_type=product_platform, sub_source_type=product_hunt, item_type=product。产品介绍放 content，tagline、makers、官网、topics 放 context_content。

V2EX / LINUX DO / Reddit

source_type=community, sub_source_type 区分平台，item_type=post。帖子正文放 content，精选回复和楼层摘要放 context_content。

明确不进最终表

字段 结论 原因
canonical_url drop 对去重有用，但会让下游同时面对两个 URL。先由八爪鱼内部决定 id 和 url。
source_item_id merge 不是所有来源都有稳定 ID。有的话放 extra，例如 tweet_id、hn_id、repo_full_name。
author_name merge 展示需要但不是核心 contract。需要时放 extra.author_name。
fetch_status / fetch_error drop 最终输出表只存成功产物，失败和重试属于中间过程。
raw_payload drop 体积和结构不可控。调试需要时放日志、对象存储或内部中间表。
source_updated_date merge 多数来源没有。少数来源如 GitHub repo updated_at 可放 extra.source_updated_date。
